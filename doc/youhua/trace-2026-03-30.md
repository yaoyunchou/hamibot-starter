## 2. 高优先级问题（影响成功率 / 稳定性）

### 2.1 `Path bounds must not be negative`（Java 异常）

- **现象**：至少 2 次与「点击进入金币页面」或 `mainPopupTask` 同时出现（如 17:02:32、18:50:15）。
- **含义**：通常为 **手势路径或点击区域坐标非法**（宽高为负、越界未裁剪等），属系统层拒绝执行。
- **优化方向**：
  1. 在封装 `click` / `gesture` / `swipe` 前统一做 **bounds 校验**（`left < right`、`top < bottom`、与屏幕交集非空）；非法时打结构化日志（元素 text、raw bounds、屏幕尺寸）并跳过或改用中心点点击。
  2. 对「金币入口」等关键点击，若来自缓存坐标，增加 **失效检测**（页面旋转、分屏、折叠屏后重算）。
  3. 与 Hamibot/Auto.js 版本差异对照，确认是否某 API 在特定 ROM 上对负值更严格。

### 2.2 `backMainPage: 死循环，关闭闲鱼` 高频触发

- **规模**：当日日志中 **约 12 次**。
- **典型前置**：
  - 停留在 `com.taobao.idlefish.webview.WebHybridActivity`（混合 Web 页），无障碍树与原生「主弹框 / 闲鱼币」策略不匹配，`maxLoopTime` 递增直至阈值。
  - 从 `goOtherApp` 返回后 **当前包名仍为支付宝** `com.eg.android.AlipayGphone`，多次「切回闲鱼」仍失败，同样触发死循环保护。
  - 偶发 `com.miui.home`、`android.view.View`、`TMSActivity` 等，说明 **回栈与前台应用判断** 与真实用户路径不一致。
- **优化方向**：
  1. **WebHybridActivity**：单独分支——优先 `back()` 次数上限内退出 H5；或结合 URL/标题检测；避免与原生 `coinExchangeMain` 查找混在同一重试环内空转。
  2. **从第三方 App 返回**：在 `goOtherApp` 结束后增加 **显式 `launch` 闲鱼包名** 或 `recentTask` 切换，并 `sleep` 等待前台稳定后再进 `backMainPage`，减少「已切回仍判非闲鱼」的竞态。
  3. **死循环阈值动作**：当前为关闭 App；可评估改为「仅上报 alert + 延长冷却 + 重新 `findPage: goldCoin`」，避免频繁杀进程导致任务状态与 UI 更难收敛（需结合任务超时策略）。

### 2.3 金币页「返回」路径误伤

- **现象**：`backMainPage: 在金币主页，执行返回` 后很快 `死循环，关闭闲鱼`（如 17:05 段）。任务行消失后仍用 `back` 退出金币页，易进入 Web 或错误层级。
- **优化方向**：区分「需要关闭 H5 子页」与「需要留在金币 Tab 内打开任务列表」；对「主弹框已关闭」场景优先 **重新打开任务中心**（已知入口文案/坐标）而非连续 `back`。

## 3. 中优先级（耗时与策略命中率）

### 3.1 `findTargetElementWithCache` 大量「全部策略未命中」

| 场景 | 示例 | 单次耗时 |
|------|------|----------|
| `browseGoodsInSpecifiedChannel` | `[点击领取]`、`[任务完成]` | 约 1.6–2.5s × 多轮 |
| `coinExchangeMain` | `[闲鱼币抵扣]` | 约 2s，极端 **8–10s** |

- **现象**：频道浏览任务中多次未命中仍最终打出「浏览完成」，说明 **结束条件与 OCR/控件策略脱节**，存在无效轮询。
- **优化方向**：
  1. 为 H5/列表页增加 **备用结束条件**（停留时长、返回后任务列表变化、`hasRun` 更新），减少固定次数滑动 + 盲目找「任务完成」。
  2. `闲鱼币抵扣` 长耗时：对 Web 页降级为 **短超时 + 快速失败**，回到原生金币页再重试，避免单次查找占满 8–10s。
  3. 统一「未命中」日志：带上 **pageKey、策略索引、是否 WebView**，便于后续做命中率统计。

### 3.2 `mainPopupFn` 匹配「去完成」耗时长

- **现象**：单次查找「去完成」可达约 **15–16 秒**，且需遍历 20+ 个按钮算与任务标题的纵向距离。
- **优化方向**：
  1. 列表内先 **scroll 使任务标题进入可视区**，再缩小「去完成」候选集合。
  2. 缓存上次成功任务的 **列表项区域** 或相对顺序，减少全表扫描。
  3. 对已知任务类型优先 **文案 + 父容器** 关联，而非纯纵坐标差值（差值为负时已出现误点第 1 个「去完成」的风险，见 18:51 段日志）。

### 3.3 `navigateToMyTab` / `goBackMyPage`

- **现象**：「未找到我的 Tab 按钮」偶发，第二次重试常成功。
- **优化方向**：首次失败后 **稍长等待 + 一次 `recents` 或重拉布局**，避免在动画未结束时查找。

### 3.4 `mainPopupTask: 未找到签到按钮`

- **规模**：约 **9 次**。
- **优化方向**：签到非阻塞时可直接跳过并打 debug 级别日志；若影响后续列表状态，再补「签到」多种文案/位置策略。

## 4. 低优先级（可维护性与观测）

### 4.1 Trace 日志去重与节流

- **问题**：同一秒内大量相同 `action`（如 `findPage: home`）使文件膨胀、不利于 diff。
- **建议**：对连续相同消息做 **计数合并**（例如「findPage: home (×30 in 1s)」），或提高该路径日志级别。

### 4.2 日志完整性

- **现象**：个别行疑似截断或与上一条拼接异常（如某行仅见 `缓存候选9 errorTime=37'`）。
- **建议**：检查多线程写同一 trace 文件时是否需 **队列单线程写入** 或文件锁。

### 4.3 与 `doc/optimization.md` 的衔接

- 历史文档中已记录 `backMainPage` / `findPage` 字符串不一致等问题，本次日志中 **WebHybridActivity** 与 **第三方 App 未切回** 为新的主要矛盾，建议在 `optimization.md` 中增加一节「混合容器与跨 App」索引到本文。

## 5. 建议落地顺序（摘要）

1. **Bounds 校验 + 关键点击防护**（消除 `Path bounds` 类崩溃）。
2. **backMainPage：WebHybrid + 支付宝未切回** 分支与 **launch 闲鱼** 兜底。
3. **browseGoodsInSpecifiedChannel / 闲鱼币抵扣** 超时与结束条件收紧。
4. **mainPopupFn** 列表滚动与候选缩小，降低单次 15s+ 查找。
5. **Trace 节流与写入一致性**（工程化，便于下次再分析）。
